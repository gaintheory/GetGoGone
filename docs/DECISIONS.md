# Decisions

Use this file to record major project choices so the product does not drift.

## 2026-05-19

- Project folder created at `C:\Users\crypt\Desktop\getgogone`.
- App scaffolded with Next.js, TypeScript, Tailwind CSS, ESLint, and App Router.
- GetGoGone will be built as a full-stack SaaS-style product, not just a Facebook ad helper.
- Carsforsale/MyCommandCenter is the current inventory and lead source, but it will be modeled as an integration rather than the product backbone.
- Spanish ad/message support is a core requirement.
- Down payment advertising is supported; payment claims need compliance guardrails.
- The Claude Design handoff is the visual source of truth. Implementation should preserve its theme, styles, spacing, colors, typography, and component feel unless a design change is explicitly approved.
- Every meaningful change to product scope, architecture, data model, integrations, design system, workflows, compliance, or implementation direction must be documented in the relevant markdown file during the same work session.
- The Claude handoff has been preserved in `prototype-handoff/` and mechanically ported into `src/prototype/` as React modules. The original handoff stylesheet is used as `src/app/globals.css` to preserve the exact visual theme.
- The existing dealership Supabase project will be treated as an integration/source system first. Current source tables include `inspections`, `service_records`, `marketing_posts`, and `credit_applications`; credit application data is considered sensitive and out of scope for general GetGoGone UI until a secure credit module is designed.
- GetGoGone-owned tables are added beside the existing dealership tables rather than replacing them. Existing `inspections` and `service_records` data will feed inventory through a mapping layer and the `inspection_vehicle_source` view.
- The GetGoGone core schema migration was applied to the linked Supabase project. Generated TypeScript database types were refreshed after the migration.
- Inventory is the first screen with a live-data seam. It fetches from `/api/inventory`, which uses the `inspection_vehicle_source` view, while preserving the original Claude demo inventory as fallback.
- Live inventory state is now lifted to the prototype app shell and passed into vehicle-dependent screens, preventing Campaign Builder, Designer, Creatives, and Test Drive from using a separate hardcoded vehicle list.
- Because the live source feed does not reliably include approved weekly/monthly payment terms, GetGoGone should prefer down-payment-first and approval-safe language rather than displaying `$0/wk` or generating payment claims.
- Until a reliable Carsforsale photo feed is available, GetGoGone supports local VIN-named images in `public/images/vehicles` and maps them onto live vehicles by VIN.
- Campaign Builder V2 will be channel-module based. Each marketing channel can define its own setup fields, generated outputs, creative formats, and publishing/export method instead of sharing one generic form.
- The Campaign Builder screen now uses the channel-module model in the UI. Selected channels are treated as configurable marketing modules, so future channels such as CarGurus, Autotrader, or Cars.com can be added or removed by updating module definitions instead of redesigning the whole builder.
- Campaign Builder now saves draft campaigns to Supabase through `/api/campaigns`. The endpoint upserts the selected vehicle by VIN, creates a `campaigns` draft, and creates one `campaign_channels` draft per selected module.
- The Campaigns list now reads saved Supabase campaigns instead of generating demo rows from inventory. Saved campaigns open into a Campaign Review screen where each channel draft can be reviewed and copied separately.
- Campaign Review channel cards are editable. Headline, copy, CTA, setup fields, and publishing path edits save back to `campaign_channels` through the campaign API.
- Campaign Review exports now generate downloadable text/checklist packages at both the full-campaign and individual-channel level.
- Campaign Review exports and manual attachments now register `campaign_assets` rows so generated packages, URL/note references, and saved Designer creatives are tracked with the campaign and channel. Saved creative attachments use `template_id`; rendered image files will use the same table once storage-backed creative export is added.
- Designer rendered PNG export now uploads to the `campaign-assets` Supabase Storage bucket through `/api/creative-exports`. The route updates saved creative previews and can create `campaign_assets` image records when campaign/channel context is supplied.
- Templates are no longer a separate top-level tab. Templates are reusable starting-point assets inside the Designer, alongside a first-class Blank Canvas option. Finished/saved outputs belong in the Creatives/asset-library area.
- Designer is now vehicle-first. The left rail opens on Vehicle, the selected vehicle photo is always the base of the canvas, and "blank" means a blank overlay on top of the vehicle photo. A Clear Overlay toolbar action removes overlay layers while preserving the vehicle photo base. A Craigslist Lead Image template was added for marketplace-style exports.
- Designer's vehicle photo base is protected. Overlay photo/collage tools can add more vehicle imagery, but the background and base vehicle photo cannot be deleted, duplicated, reordered, dragged, resized, or edited.
- Designer now includes a Stack panel and full layer controls for duplicate, reorder, lock/unlock, delete, undo, redo, reset, and clear overlay. Shape/layout tools provide common ad-building blocks without introducing a separate template-builder workflow.
- Designer now has an Offer tab for down payment, price, weekly/monthly fallbacks, and disclosure. Template variables use safe fallback labels instead of rendering `$0 Down`; weekly/monthly numbers require explicit approval toggles.
- Designer now has an Import tab for template source intake. Imported third-party designs require clear commercial-use rights, and official partner badges/logos remain upload-only with permission.
- Blank overlay creation is richer: Text now includes reusable building blocks such as vehicle title, offer footer, Spanish CTA, and marketplace info panel.
- Designer save actions are no longer placeholder toasts. Save template persists a reusable `creative_templates` row with `category = custom_template`; Save version persists `category = saved_creative`. Saved templates reload into Designer > Templates, and saved creatives reload into Designer > Saved and the Creatives asset library.
- Product direction is now agency-owner first: GetGoGone is being built as a personal multi-client marketing command center operated by the agency owner, not primarily as a salesperson self-serve SaaS. Salespeople may receive tasks/leads later, but campaign planning, AI generation, creative production, publishing, and reporting are centered on the operator.
- `docs/AGENCY_ROADMAP.md` is the master roadmap for the agency command center direction. Roadmap schema/API entries are candidates until each phase gets a focused migration/implementation plan.
- Agency Foundation implementation has started. The app now opens to an Agency Command screen, loads clients from `/api/agency/clients`, stores `activeClientId` in local storage, and scopes inventory, campaign, creative-template, Designer, and Creatives API calls to the active client.
- For now, one `dealerships` row equals one agency client. The default dealership continues to use the existing `inspection_vehicle_source` feed so current Right Price inventory stays intact while future clients can use normalized `vehicles` rows.
- Marketing Cockpit implementation has started as a derived queue, not a persisted task table. `/api/agency/cockpit` scans inventory, campaigns, and saved creatives to flag missing offers, missing usable photos, missing first campaigns, missing Spanish campaigns, missing saved creatives, and draft campaigns needing review.
- Brand Brain implementation has started. `client_brand_brains` is now the client-specific source for tone, Spanish adaptation, approved/banned phrases, down-payment rules, finance disclaimers, audience notes, objection handling, and future AI/compliance generation context.
- AI Provider Layer implementation has started with local Ollama status and generation routes. Campaign Builder now uses `/api/ai/generate-copy` for per-channel AI copy, reading Brand Brain rules and falling back to compliant deterministic copy when local AI is offline.
- Local AI should use all available Ollama models intelligently. The provider now discovers installed models and selects by task rather than hardcoding one model for every generation.
- Generated campaign copy is saved to `generated_outputs` with provider, model, prompt context, language, target channel, and draft status. The generator also runs a deterministic cleanup pass so local model output cannot introduce risky payment-today, drive-away-today, or missing-disclaimer language.
- Campaign Builder and campaign previews must use safe down-payment fallbacks when approved terms are missing. They should show "Down payment options available", "Options", or "Terms" instead of rendering `$0 down`, `$0/wk`, or universal financing claims.

## Phase 0.1 — Site auth gate (2026-05-23)

- Chose a single shared password over Google OAuth + per-user allowlist (Option 5 from earlier auth-options discussion). Rationale: solo operator use today with occasional owner peek; no per-user identity needed yet. Upgrade path is documented — swapping to Supabase Auth + Google OAuth + allowlist table later requires replacing only the login page and the middleware check; the signed-cookie helper survives the migration.
- Implementation: HMAC-SHA256 signed `ggg_auth` cookie (Web Crypto API, constant-time comparison, 30-day expiry). Cookie is httpOnly, sameSite=lax, secure in production. Edge middleware gates every request; unauthenticated pages redirect to `/login?next=<original>`, unauthenticated API calls return 401.
- Public prefixes whitelisted in middleware: `/v/*` (per-vehicle landing pages) and `/api/leads/inbound` (web inquiry endpoint). These must remain accessible without the cookie so web inquiry attribution can work end-to-end.
- New env vars: `SITE_PASSWORD`, `SITE_AUTH_COOKIE_SECRET` (32-byte hex HMAC key), `PUBLIC_BASE_URL`.
